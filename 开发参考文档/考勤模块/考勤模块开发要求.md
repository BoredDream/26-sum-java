# 要求3：学生考勤管理模块开发要求

> 文档版本：V1.0  
> 适用项目：《基于 Java 的综合实训选题系统》  
> 对应指导书条目：第 3 条「学生考勤管理。包括在线签到、文档提交等。」  
> 说明：根据项目统一规范，**文档提交/过程文档/开发日志统一归入选题管理模块**，考勤模块仅负责签到、补签、统计、导出。  
> 模块负责人：范继青  
> 后端包路径：`com.training.system.attendance`

---

## 1. 模块定位

学生考勤管理模块聚焦综合实训期间的出勤过程管理，为成绩评估模块提供考勤原始数据。模块必须严格遵守以下边界：

- **属于本模块**：签到任务发布、在线签到、补签申请、补签审核、考勤记录查询、考勤统计、考勤报表导出。
- **不属于本模块**：过程文档上传、过程文档审核、开发日志管理（归入选题管理模块 `com.training.system.selection`）。
- **数据原则**：本模块产生并维护 `attendance_task`、`attendance_record`、`makeup_sign_apply` 三张表，可被成绩模块读取，但成绩模块不得反向修改原始签到记录。

## 4. 功能详细要求

### 4.1 发布签到任务（F01）

#### 4.1.1 输入数据

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| task_title | string | 是 | 签到任务标题，如「第3周周一上午签到」。 |
| task_type | int | 是 | 1-日常签到，2-阶段签到。 |
| scope_type | int | 是 | 1-班级，2-团队，3-全部。 |
| scope_value | string | 条件必填 | `scope_type=1` 填班级名称；`scope_type=2` 填团队ID；`scope_type=3` 可空。 |
| start_time | datetime | 是 | 签到开始时间。 |
| end_time | datetime | 是 | 签到结束时间，必须晚于开始时间。 |
| description | string | 否 | 签到说明。 |

#### 4.1.2 业务规则

1. 只有教师或管理员可发布签到任务。
2. `end_time` 必须晚于 `start_time`。
3. `scope_type=2` 时，`scope_value` 必须是已存在的 `team_id`。
4. 任务发布后，系统应为适用范围内的每一名学生预生成一条 `attendance_record`，初始 `sign_status=0`（缺勤）。
5. 任务发布者记录为 `attendance_task.teacher_id`。

#### 4.1.3 异常情况

| 异常情况 | 系统处理 |
| --- | --- |
| 标题为空 | 提示「请输入签到标题」。 |
| 结束时间早于开始时间 | 提示「结束时间必须晚于开始时间」。 |
| 未选择适用范围 | 提示「请至少选择一个班级或团队」。 |
| 团队ID不存在 | 提示「适用范围中的团队不存在」。 |
| 非教师/管理员操作 | 返回 `403 无权限`。 |

---

### 4.2 在线签到（F02）

#### 4.2.1 输入数据

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| task_id | long | 是 | 签到任务ID。 |
| remark | string | 否 | 签到备注。 |

> 学生ID从当前登录 Session 获取，不由前端传入。

#### 4.2.2 业务规则

1. 只有学生可签到。
2. 学生只能签到属于本人范围内的任务：
   - `scope_type=1`：学生 `class_name` 与 `scope_value` 一致；
   - `scope_type=2`：学生属于 `scope_value` 对应的团队；
   - `scope_type=3`：全部学生可签到。
3. 当前时间必须在 `[start_time, end_time]` 内。
4. 每个学生对同一任务只能签到一次（`sign_status` 已变为 1/2/3 时不可重复签到）。
5. 签到状态判定：
   - 当前时间 ≤ `start_time + 15 分钟`：`sign_status=1`（正常）。
   - `start_time + 15 分钟` < 当前时间 ≤ `end_time`：`sign_status=2`（迟到）。
6. 签到成功后更新 `attendance_record.sign_time`、`sign_status`、`remark`。

#### 4.2.3 异常情况

| 异常情况 | 系统处理 |
| --- | --- |
| 不在签到时间范围内 | 提示「当前不在签到时间内，请在规定时间内签到」。 |
| 已完成签到 | 提示「您已完成本次签到，请勿重复签到」。 |
| 无有效签到任务 | 提示「当前没有进行中的签到任务」。 |
| 非学生操作 | 返回 `403 无权限`。 |
| 学生不在任务适用范围内 | 提示「您不在本次签到范围内」。 |

### 4.3 补签申请（F03）

#### 4.3.1 输入数据

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| task_id | long | 是 | 需要补签的签到任务ID。 |
| record_id | long | 否 | 关联的考勤记录ID，为空时后端根据 task_id + student_id 查找。 |
| apply_reason | string | 是 | 补签原因，长度不少于 5 个字符。 |
| proof_file_path | string | 否 | 证明材料文件路径。 |

#### 4.3.2 业务规则

1. 只有学生可申请补签。
2. 学生只能为本人考勤记录申请补签。
3. 同一学生对同一任务只能存在一条「待审核」补签申请。
4. 可申请补签的条件：对应考勤记录 `sign_status` 为 0（缺勤）或 2（迟到）。
5. 证明材料文件大小不超过 50MB，允许格式：`doc, docx, pdf, png, jpg, jpeg`。

#### 4.3.3 异常情况

| 异常情况 | 系统处理 |
| --- | --- |
| 重复提交补签申请 | 提示「该签到任务已有待审核补签申请」。 |
| 非本人考勤记录 | 提示「只能为本人考勤记录申请补签」。 |
| 考勤记录已正常 | 提示「当前考勤记录无需补签」。 |
| 补签原因为空或过短 | 提示「请填写补签原因，至少 5 个字符」。 |
| 证明材料格式/大小不符 | 按统一文件上传规范提示。 |

---

### 4.4 补签审核（F04）

#### 4.4.1 输入数据

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| apply_id | long | 是 | 补签申请ID。 |
| audit_status | int | 是 | 1-通过，2-驳回。 |
| audit_comment | string | 驳回时必填 | 审核意见。 |

> 审核教师ID从当前登录 Session 获取。

#### 4.4.2 业务规则

1. 只有教师或管理员可审核补签申请。
2. 教师只能审核本人发布的签到任务对应的补签申请；管理员可审核全部。
3. 驳回时必须填写审核意见。
4. 审核通过后：
   - 更新 `makeup_sign_apply.audit_status=1`；
   - 更新对应 `attendance_record.sign_status=3`（补签），`is_makeup=1`。
5. 审核驳回后：
   - 更新 `makeup_sign_apply.audit_status=2`；
   - 保持 `attendance_record` 原状态不变。
6. 已审核的申请不可再次审核。

#### 4.4.3 异常情况

| 异常情况 | 系统处理 |
| --- | --- |
| 无权限审核 | 提示「无补签审核权限」。 |
| 审核意见为空 | 提示「请填写审核意见」。 |
| 申请已被处理 | 提示「该申请已审核，不能重复操作」。 |

---

### 4.5 考勤记录查询（F05）

#### 4.5.1 查询条件

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| task_id | long | 按签到任务筛选。 |
| student_id | long | 按学生筛选（教师/管理员可用）。 |
| sign_status | int | 0-缺勤，1-正常，2-迟到，3-补签。 |
| start_date | date | 签到时间范围开始。 |
| end_date | date | 签到时间范围结束。 |
| keyword | string | 按任务标题、学生姓名模糊查询。 |

#### 4.5.2 权限规则

| 角色 | 可见范围 |
| --- | --- |
| 学生 | 仅本人考勤记录。 |
| 教师 | 本人发布的签到任务下的所有记录。 |
| 管理员 | 全部记录。 |

---

### 4.6 考勤统计（F06）

#### 4.6.1 统计维度

| 维度 | 说明 |
| --- | --- |
| 按任务统计 | 某任务下应签人数、实签人数、正常、迟到、缺勤、补签数量及出勤率。 |
| 按学生统计 | 某学生在指定时间范围内的出勤率、迟到次数、缺勤次数、补签次数。 |
| 按班级/团队统计 | 某班级或团队在指定时间范围内的整体出勤情况。 |

#### 4.6.2 计算公式

- 出勤率 = (正常签到次数 + 补签次数) / 应签总次数 × 100%。
- 应签总次数 = 该范围内任务数 × 该范围内学生数（按任务维度统计时）。

### 4.7 考勤报表导出（F07）

#### 4.7.1 导出内容

| 列名 | 说明 |
| --- | --- |
| 任务标题 | 签到任务名称。 |
| 学生姓名 | 学生姓名。 |
| 学号 | 学生学号。 |
| 班级 | 学生班级。 |
| 签到时间 | 实际签到时间。 |
| 签到状态 | 正常/迟到/缺勤/补签。 |
| 是否补签 | 是/否。 |
| 备注 | 签到备注或补签原因。 |

#### 4.7.2 业务规则

1. 仅教师、管理员可导出。
2. 教师只能导出本人发布的任务数据。
3. 导出格式为 Excel（.xlsx）。
4. 大数据量时支持按任务或时间范围分批导出。

## 10. 测试用例要求

每个功能必须覆盖以下四类用例：

### 10.1 正常用例

| 编号 | 功能 | 输入 | 预期结果 |
| --- | --- | --- | --- |
| T01 | 正常发布签到任务 | 填写完整任务信息，范围选择班级。 | 任务发布成功，范围内学生生成缺勤记录。 |
| T02 | 正常签到（正常状态） | 学生在开始时间后 5 分钟内签到。 | 签到成功，状态为「正常」。 |
| T03 | 迟到签到 | 学生在开始时间后 20 分钟签到。 | 签到成功，状态为「迟到」。 |
| T04 | 正常补签并通过 | 学生缺勤后提交补签申请，教师审核通过。 | 考勤记录状态更新为「补签」。 |

### 10.2 异常用例

| 编号 | 功能 | 输入 | 预期结果 |
| --- | --- | --- | --- |
| T05 | 结束时间早于开始时间 | 设置结束时间 < 开始时间。 | 提示「结束时间必须晚于开始时间」。 |
| T06 | 重复签到 | 学生已签到后再次签到。 | 提示「您已完成本次签到，请勿重复签到」。 |
| T07 | 超时签到 | 签到结束后尝试签到。 | 提示「当前不在签到时间内」。 |
| T08 | 重复补签申请 | 对同一任务重复提交待审核补签申请。 | 提示「该签到任务已有待审核补签申请」。 |

### 10.3 权限用例

| 编号 | 功能 | 输入 | 预期结果 |
| --- | --- | --- | --- |
| T09 | 学生发布签到任务 | 学生账号调用 POST `/api/attendance/task`。 | 返回 `403 无权限`。 |
| T10 | 教师审核他人任务补签 | 教师账号审核非本人发布任务的补签申请。 | 返回 `403 无权限」。 |
| T11 | 学生查看他人考勤 | 学生账号查询其他同学考勤记录。 | 返回 `403 无权限` 或仅返回本人数据。 |

### 10.4 边界用例

| 编号 | 功能 | 输入 | 预期结果 |
| --- | --- | --- | --- |
| T12 | 正好第15分钟签到 | 学生在 `start_time + 15 分钟` 整点签到。 | 状态为「正常」。 |
| T13 | 第15分钟零1秒签到 | 学生在 `start_time + 15 分钟 + 1 秒` 签到。 | 状态为「迟到」。 |
| T14 | 空范围任务 | 适用范围班级不存在学生。 | 任务可发布，但应签人数为 0，统计时不出错。 |

## 11. 非功能性要求

1. **性能**：签到接口响应时间 ≤ 500ms；统计接口响应时间 ≤ 2s。
2. **并发**：同一任务高并发签到时，通过数据库唯一约束（`uk_attendance_task_student`）防止重复签到。
3. **安全**：文件上传必须校验类型和大小，禁止可执行文件；证明材料路径不得暴露服务器绝对路径。
4. **日志**：关键操作（发布任务、签到、审核补签）建议记录操作日志（可由信息管理模块统一实现，本模块调用）。
5. **兼容性**：符合项目统一技术栈：Java 18、Spring Boot 3.3.5、MyBatis、MySQL 8、Maven。

## 13. 开发前确认事项

正式编码前，模块负责人必须与项目负责人确认：

1. 教师与班级/学生的对应关系如何确定？当前 schema 未建立 `teacher-class` 关联，本模块暂定「教师只能查看/审核本人发布的任务数据」。
2. 签到任务状态（未开始/进行中/已结束）是实时计算还是通过定时任务/操作触发更新？
3. 补签申请是否设置时限（如签到结束后 24 小时内）？
4. 考勤统计周期默认按整个实训周期、周、还是月？
5. 报表导出是否需要支持按班级/团队批量导出？
