## 1. 文档目的

本文件用于统一《基于 Java 的综合实训选题系统》的开发规范。所有成员在正式编码前，必须先阅读并遵守本文档。

本文档解决以下问题：

1. 避免每个人使用不同项目结构，导致后期无法合并。
2. 避免重复开发相同功能，例如文档上传、用户登录、权限判断。
3. 避免数据库字段命名不一致，导致接口和 SQL 难以联调。
4. 统一 GitHub 分支、提交、Pull Request 和代码合并规则。
5. 明确每个成员在开发前需要补充的接口清单、页面清单和模块设计说明。

本文件不是最终数据库设计文档，也不是最终接口文档。数据库总表设计、接口清单和页面清单会在后续继续补充。

## 3.2 IDEA 和 VS Code 是否有影响

允许成员使用不同开发工具：

| 工具 | 是否允许 | 要求 |
| --- | --- | --- |
| IntelliJ IDEA | 允许 | 通过 Maven 导入后端项目，不提交个人 IDE 配置 |
| VS Code | 允许 | 安装 Java、Spring Boot、Maven、Vue 相关插件，不改变项目结构 |

结论：

> IDEA 和 VS Code 不会影响协作，前提是所有人使用统一的 Maven 项目结构、JDK 18、统一 GitHub 分支规则和统一代码规范。

1. 不允许因为个人 IDE 改动项目目录结构。
2. 不允许提交 `target/`、`node_modules/`、本地日志文件等无关内容。
3. 不允许把本地数据库账号密码、API Key 写死后提交到 GitHub。
4. 不允许绕过 Maven 手动导入 jar 包。

## 5. 后端项目结构规范

Spring Boot 后端建议采用以下包结构：

```text
com.training.system
├── common                   # 通用返回类、分页类、常量
├── config                   # 配置类，如跨域、拦截器、MyBatis 配置
├── controller               # 控制层，接收前端请求
├── service                  # 业务接口
├── service.impl             # 业务实现类
├── mapper                   # MyBatis Mapper 接口
├── entity                   # 数据库实体类
├── dto                      # 请求参数对象
├── vo                       # 返回给前端的视图对象
├── enums                    # 状态枚举
├── exception                # 全局异常处理
└── utils                    # 工具类
```

### 5.1 各层职责

| 层 | 职责 | 注意事项 |
| --- | --- | --- |
| controller | 接收请求、参数校验、返回结果 | 不写复杂业务逻辑 |
| service | 编写业务规则 | 业务逻辑主要写在这里 |
| mapper | 数据库访问 | 不写业务判断 |
| entity | 数据库表映射 | 字段应尽量和数据库一致 |
| dto | 接收前端请求参数 | 新增、修改、查询建议分别建 DTO |
| vo | 返回前端展示数据 | 避免直接把数据库敏感字段返回给前端 |
| common | 通用类 | 全组统一使用，不要各模块重复造 |

## 6. 前端项目结构规范

Vue 前端建议采用以下结构：

```text
frontend/
├── src/
│   ├── api/                 # 后端接口封装
│   ├── assets/              # 静态资源
│   ├── components/          # 公共组件
│   ├── router/              # 路由配置
│   ├── stores/              # 状态管理，可选
│   ├── utils/               # 工具函数
│   ├── views/               # 页面
│   │   ├── topic/           # 出题管理页面
│   │   ├── selection/       # 选题管理页面
│   │   ├── attendance/      # 考勤管理页面
│   │   ├── info/            # 信息管理页面
│   │   └── score/           # 成绩评估页面
│   ├── App.vue
│   └── main.js
├── package.json
└── vite.config.js
```

### 6.1 前端开发要求

1. 每个模块页面放入对应的 `views` 子目录。
2. 每个模块接口统一放入 `src/api/模块名.js`。
3. 不要在页面里直接写完整接口地址，应通过统一的 axios 工具封装。
4. 表格、分页、按钮、弹窗风格尽量统一。
5. 页面权限应配合后端权限，前端只负责隐藏按钮，真正权限必须由后端校验。

## 9. 统一业务规则

| 编号 | 规则 |
| --- | --- |
| R1 | 一个学生在同一实训周期内只能加入一个团队。 |
| R2 | 一个团队同一时间只能存在一个待审核选题申请。 |
| R3 | 一个题目可以被多个团队选择。 |
| R4 | `student_limit` 表示单个团队人数上限。 |
| R5 | 选题审核通过后，是否允许换题由选题模块设计者进一步细化，但必须保留操作记录。 |
| R6 | 过程文档与开发日志统一归入选题管理模块。 |
| R7 | 考勤管理模块不重复实现文档上传和文档审核。 |
| R8 | 用户登录账号、密码、角色统一存储在 `user_account` 表中。 |
| R9 | 学生表、教师表不再存储 `password` 字段。 |
| R10 | 成绩模块统一命名为“成绩评估与过程分析模块”。 |
| R11 | AI 分析报告学生可见。 |
| R12 | 成绩同时保留团队成绩和个人成绩设计空间，团队成绩主表使用 `score`，个人成绩表使用 `student_score`，具体计算规则由成绩模块开发者细化后提交项目负责人确认。 |
| R13 | 代码过程数据使用 Git 仓库作为来源。 |
| R14 | 题目、文件、公告等业务数据采用逻辑删除。 |
| R15 | 数据库总表结构由项目负责人汇总设计，各模块不得私自新增核心公共表。 |

| 字段名 | 小写下划线 | `student_id` |

## 10.6 题目表补充字段

由于本项目规定一个题目可以被多个团队选择，因此题目表中需要明确以下字段含义：

| 字段名 | 说明 |
| --- | --- |
| student_limit | 单个团队人数上限 |
| selection_start_time | 选题开始时间 |
| selection_end_time | 选题截止时间 |
| team_limit | 允许选择该题目的最大团队数，可选 |
| selected_team_count | 当前已选择该题目的团队数量，可选 |
| is_full | 是否已满，可选 |

说明：

1. 如果不限制一个题目最多多少团队选择，可以暂不使用 `team_limit`。
2. 如果需要展示题目是否已满，可以通过 `team_limit` 和 `selected_team_count` 判断。
3. `student_limit` 不是该题目的总学生人数上限，而是每个团队的人数上限。

| 分页查询 | GET | `GET /api/topic/page` |
| 审核 | POST | `POST /api/topic/{topicId}/review` |
| 提交 | POST | `POST /api/topic/{topicId}/submit` |

## 14. 权限设计规范

## 14.1 基础角色

系统统一角色：

| 角色 | 权限范围 |
| --- | --- |
| 学生 | 只能操作本人或本团队相关数据 |
| 教师 | 只能管理本人发布题目、指导团队和相关成绩 |
| 管理员 | 进行系统级数据维护、异常处理和备份 |

---

## 14.2 权限判断原则

1. 前端可以隐藏无权限按钮，但不能代替后端权限校验。
2. 后端接口必须判断当前登录用户角色。
3. 涉及本人、本团队、本人指导团队的数据，必须判断数据归属。
4. 删除、审核、评分、备份等关键操作必须严格限制权限。
5. 管理员操作也应记录操作日志。

## 15. 登录与鉴权建议

本项目建议先采用简单稳定的方式：

```text
Session 登录 + 后端拦截器权限校验
```

原因：

1. 实现难度低。
2. 适合课程实训项目。
3. 比 JWT 和完整 Spring Security 更容易调试。
4. 能满足学生、教师、管理员三类角色权限控制。

若后续有能力，可以再升级为 Spring Security 或 JWT。

## 20. 后端开发要求

1. Controller 不直接写 SQL。
2. Controller 不写复杂业务逻辑。
3. Service 层负责业务判断。
4. Mapper 层只负责数据库访问。
5. 多表写入或状态流转需要使用事务。
6. 所有接口统一返回 `Result` 格式。
7. 异常统一交给全局异常处理器。
8. 不使用 `System.out.println` 作为正式日志。
9. 不允许在代码中硬编码数据库密码、AI Key、GitHub Token。
10. 涉及权限的数据操作必须判断当前登录用户角色和数据归属。

---

## 21. 前端开发要求

1. 不在页面中直接拼接后端完整地址。
2. 接口请求统一放到 `src/api`。
3. 页面必须做基本表单校验。
4. 后端返回错误时，前端应显示错误提示。
5. 列表页建议统一支持分页、查询和重置。
6. 重要操作如删除、审核、提交成绩，应有二次确认。
7. 权限按钮根据当前用户角色显示或隐藏。
8. 不在前端存储敏感密码。
